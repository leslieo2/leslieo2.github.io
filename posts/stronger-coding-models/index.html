强模型真正改变了什么

这两年的 AI 编程圈，多少有点进入了一种“强模型狂热”。从 Anthropic 的 Sonnet 4 到 Opus 4.6，从 OpenAI 的 GPT-5.1-Codex 到 GPT-5.4，各家的发布叙事几乎一致：更强的 coding、更长的 agentic work、更好的 code review、更可靠的 bug fixing。这套说法当然成立，因为模型越强，的确越能吞掉软件实现中的大量偶然复杂性，样板代码、框架胶水、测试脚手架、常规修补都能更高效地自动完成。但问题也恰恰从这里开始。如果一个编程模型已经足够强，强到能处理大部分实现层面的负担，那么我们继续追求“更强”，到底在追求什么？更进一步说，问题并不是还要不要更强模型，而是“更强”到底应该主要体现在代码补全能力上，还是体现在获取约束、验证结果和控制演化的能力上。

为什么软件会越来越同质化

如果把软件开发简化成一次信息分工，这个问题会更清楚一些。我们可以先用一个启发式表达来理解：在某个阶段的软件产物里，人类显式提供的信息量是 p，模型自动补全的信息量是 c。这个表达不是在假设总信息量守恒，只是为了说明一个事实：模型越强，通常就越能替人承担原本需要显式写出来的那部分信息。更严谨地说，p 和 c 都不是单一类型的信息，它们内部都可能同时包含本质信息和偶然信息。也就是说，人类显式提供的信息里，既可能有真正决定产品独特性的部分，也可能有样板、模板、胶水代码这类偶然复杂性；模型自动补全的信息里，同样既可能是在帮你消化这些偶然信息，也可能开始替你做本该由人明确决定的本质选择。

问题的关键，不是 c 变大本身，而是 c 在什么条件下变大，以及它替代的到底是哪一类信息。真实世界里，强模型一边替代部分信息输入，一边又创造新的可表达空间：它让你更快试错，于是需求空间变大；它让更多边界条件被显式化，于是约束变多；它也让原本懒得做的自动化、个性化和分层抽象进入可行区。所以真正值得警惕的，不是“模型补得越来越多”，而是在人没有把目标、边界、偏好和 tradeoff 说清楚的时候，模型补进去的已经不只是偶然信息，而开始侵入本质信息的决策空间。

一旦进入这种状态，问题才会从“分工变化”转向“产物趋同”。当输入约束不足时，模型输出会越来越受统计先验支配；而模型越强，这种基于先验的补全往往越平滑、越稳定、也越像标准答案。所以同质化并不是因为模型太弱，而是因为“约束不足 + 强先验”这个组合成立了。你没把真正重要的约束说清楚，模型就会用它见过最多、最像最佳实践的方案替你填空，最后产物看起来都“不差”，却也越来越像别人做过的东西。在不需要差异化的场景里，这未必是坏事；但在真正依赖产品立场、组织经验和策略取舍的场景里，它很难形成壁垒。

模型该吃掉的是偶然信息，不是本质信息

这也是为什么我觉得，编程模型真正该吃掉的，应该是 偶然信息（accidental information），而不是 本质信息（essential information）。前者是那些不决定产品灵魂、却大量消耗时间和注意力的东西，比如样板代码、CRUD、机械性 bug 修复、常见工程模板；后者才是真正决定一个软件独特性的部分，比如你到底在为谁优化、愿意牺牲什么保留什么、系统更重一致性还是更重响应、哪些边界 case 在你这里是核心而不是噪声。过去几十年软件工程的主线，其实一直都是不断吞掉 accidental information：汇编被高级语言吞掉，资源细节被运行时吞掉，基础设施被云平台吞掉，界面和交互的一部分被框架吞掉，今天轮到样板实现和局部推理被模型吞掉。这完全合理，也值得继续推进。但一旦我们开始期待模型替我们做 essential information 的决定，事情就变质了，因为那不再是在消除偶然复杂性，而是在用统计先验取代人的真实意图。

哪些同质化是坏事，哪些不是

不过，“同质化”这件事本身也不能讲得太满。很多软件原本就应该同质化。后台管理、权限系统、支付流程、日志审计、监控告警、企业内网工具，用户往往根本不在乎你有没有独特灵魂，只在乎它稳不稳、可不可审计、能不能维护、和已有系统接得顺不顺。在这些领域里，同质化不是缺陷，反而是工业化红利。

真正值得警惕的，不是代码表面的趋同，而是目标函数、默认策略和 tradeoff 的趋同。软件的独特性，很多时候并不主要住在代码长什么样，而住在你选了什么用户问题、定义了什么成功指标、如何设计反馈闭环、哪些默认值不能自动拍板。换句话说，代码方案趋同未必等于软件价值趋同，但如果连这些本质信息也被强先验一并抹平，那才是真正的问题。

更会写代码的边际收益为何递减

从这个角度看，所谓“更会写代码”的边际收益递减，只在一部分任务上成立。对普通 CRUD、常规修补、样板集成来说，模型从 85 分进步到 90 分，边际收益确实会越来越小；但对大仓库改动、深层回归定位、复杂迁移和高可信修复这类任务，能力一旦跨过某个可靠性阈值，收益往往是跳变而不是递减。

真正的问题在于，模型越强，不只是更会写代码，也越可能在约束不足时替你做决定。它一方面能继续压缩实现成本，另一方面也更容易侵入本质信息，把原本应该由人拍板的目标、边界和 tradeoff 一并补完。所以当 accidental information 被大量自动化之后，软件工程的主瓶颈并不会消失，而是更明确地转向“构造约束、验证正确性、整合上下文和管理演化”。

下一阶段应押注什么

因此，真正有价值的下一代模型和agent系统，至少要在四件事上明显变强。第一，是约束恢复：能从需求文档、工单、历史提交、测试和事故复盘里恢复出真正的边界条件，而不只盯着当前 prompt。第二，是分歧显式化：能在多个合理方案之间指出差别，告诉你哪些默认值牵涉成本、风险或组织责任，而不是静默地选一个最主流的答案。第三，是验证闭环：不只是生成代码，还要能把结果送进测试、静态分析、变更影响分析、沙盒回放甚至 shadow traffic 里验证。第四，是受控演化：每次改动都知道变更边界、审批路径、回滚方式和审计责任，让 agent 不只是会写第一版，而是能在真实工程里持续工作。

说到底，我们真正需要的“更强”，不是更会猜，而是更知道哪里不该猜；不是更像一个高效的平均答案制造机，而是更像一个能保留人类真实意图、把模糊目标转成受控演化的协作者。

我们还需要更强的编程模型吗

所以，对“当 Opus 4.6、GPT-5.4 轮番登场之后，我们还需不需要更强的编程模型”这个问题，我的答案仍然是：需要。但需要被反对的，从来不是强模型本身，而是拿强模型代替明确意图。如果更强只是意味着更会补全、更会 debug、更会套用主流架构、更会用行业平均最佳实践把空白补满，那么当 accidental information 已经被吃掉大半之后，这条路会越来越不够，而且它越成功，就越容易把默认答案写得平滑却平庸。

但如果我们说的“更强”，指的是更擅长发现你哪里没说清、识别哪个默认值不能替你拍板、把模糊目标转成可验证约束、用测试和历史行为去校准真实意图、并在长周期系统演化中保持一致性，那么这条路不仅没有到头，反而才刚开始。真正稀缺的，未必是一个更强的代码生成器，而是一个更强的约束解释器、验证协作者和演化代理。而要做到这些，它仍然需要更强的模型作为底座。
